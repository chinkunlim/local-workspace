# 任務：Antigravity 除錯與清理計畫 (Purge & Fix Plan)

## 【區塊 A：Bug 根因分析與修復策略】

### 1. I/O 路由異常分析 (`inbox_daemon.py`)
- **判斷機制**：`inbox_daemon.py` 目前完全依賴「副檔名」(`ext = os.path.splitext(filename)[1].lower()`) 來判斷檔案類型，並沒有深度驗證 MIME type。
- **竄軌與污染原因**：
  - 在第 116-136 行，存在 `pdf_routing_rules` 的硬編碼繞道邏輯。當 PDF 名稱符合特定條件時，會觸發 `routing_mode = "audio_ref"` 或是 `"both"`。
  - 在這個邏輯下，Daemon 會將 `target_skill` 設為 `audio-transcriber`，並且**直接將 PDF 移動至 `audio-transcriber` 的 `output/00_glossary/` 目錄**。
  - **這嚴重違反了架構鐵律**：Daemon 只應負責將檔案派發至各 Skill 的 `input/` 沙箱，`output/` 目錄只能由各個 Skill 內部專屬的 `AtomicWriter` 寫入。這種越權行為導致了資料流竄軌。
  - **為何觸發 Audio pipeline**：雖然寫入 `output` 理論上不會觸發，但在 Daemon 後續觸發機制 (`if os.sep + "input" + os.sep in target_dir + os.sep`) 的判斷中，如果您的外部路徑本身包含 `input`，就會意外發送觸發訊號給 WebUI 或 subprocess，導致 Audio pipeline 收到了自己無法處理的 PDF。
- **修復策略**：
  - 徹底移除 `inbox_daemon.py` 中這段針對 PDF 越界操作的 `pdf_routing_rules`。
  - 嚴格落實一對一沙箱派發：`.pdf` **永遠只能** 進入 `doc-parser/input/`，不准再越界寫入其他 Skill 的輸出區。

- **`config/config.yaml` 刪除清單**：
  - 🗑️ `paths.phases` 下的 `p4` (highlighted) 與 `p5` (notion_synthesis) 路徑。
  - 🗑️ `phase4` 與 `phase5` 的完整執行設定區段。
  - 🗑️ `thresholds` 與 `context` 中相關的參數 (`phase4_verbatim`, `phase5_mermaid_retry_max`)。

**針對 `doc-parser`：**

  - 🗑️ `Phase 3 Map: Concept Extraction`

### 3. Prompt 跨層安全搬移與整合 (Prompt Migration)
