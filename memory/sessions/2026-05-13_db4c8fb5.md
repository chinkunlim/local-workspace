## 1. Implementation Plan

# MarkItDown 整合計畫 — OpenClaw doc_parser 擴展

## 背景與目標

**現狀：** `doc_parser` 只接受 `.pdf` 和 `.png` 作為輸入格式。  
**目標：** 整合 Microsoft MarkItDown，讓 `.pptx` / `.docx` / `.xlsx` 成為合法 inbox 輸入格式，統一流入現有 pipeline。

**定位確認（已與用戶對齊）：**
- PPTX 備注（Speaker Notes）= 原材料，與投影片文字一同輸出為 `_raw_extracted.md`
- 備注內容作為 `audio_transcriber` 的 `with_reference` 參考資料，**不取代** audio 轉錄
- 整合後 `proofreader` pipeline 完全不需要改動

---

## Open Questions

> [!IMPORTANT]
> 以下問題請確認後再開始實作：

1. **`.pptx` 是否需要沿用現有 `pdf_routing_rules` 模式？** 例如 `L02_slides.pptx` 按 `_slide` 路由到 `audio_ref`，還是單獨定義 PPTX 路由規則？
2. **`.docx` 有無優先需求？** 目前課程材料是否有 Word 文件需要處理？
3. **MarkItDown 的圖片處理**：PPTX 裡的圖片只需要 LLM 文字描述，還是也要像 PDF 一樣提取成獨立 `assets/` 圖片檔？（建議：僅 alt-text 描述，不提取圖片，降低複雜度）

---

## 架構決策

**ADR 提案：整合 MarkItDown 作為 doc_parser 前置轉換層（p00c）**

- 新 Phase `p00c_markitdown` 在 Phase 0a 診斷之前執行
- 偵測非 PDF/PNG 副檔名 → 呼叫 MarkItDown → 輸出 `{file_id}_raw_extracted.md`（與 p01a_engine 介面完全相同）
- Phase 0a/1a/1b/1c/1d 對 PPTX/DOCX/XLSX 來源的檔案全部設為 `⏭️`（不適用）
- PPTX 備注以 `### Notes:` 區塊保留在 raw_extracted.md 中

### doc_parser 核心層

#### [NEW] skills/doc_parser/scripts/phases/p00c_markitdown.py
新 Phase，核心邏輯：

```
輸入：data/doc_parser/input/{subject}/{filename}.pptx/.docx/.xlsx
輸出：data/doc_parser/output/01_processed/{subject}/{file_id}/{file_id}_raw_extracted.md
副產物：figure_list.md（若有圖片，僅記錄位置與 alt-text，不提取圖片檔）

流程：
1. 掃描 input/ 中的 .pptx/.docx/.xlsx
2. 呼叫 MarkItDown(...).convert(filepath)
3. 後處理：
   - PPTX：保留 `<!-- Slide number: N -->` 結構 + `### Notes:` 備注區塊
   - DOCX：保留標題/段落層級
   - XLSX：保留 Markdown 表格格式
4. 輸出 {file_id}_raw_extracted.md（IMMUTABLE，帶 <!-- header -->）
5. 生成空 figure_list.md（佔位，供後續 Phase 讀取）
6. StateManager.update_task(..., "p0c", "✅")
```

Phase 的 DAG 遮罩邏輯（在 `run()` 裡手動注入）：
- `p0c` 完成後，`p0a`, `p0b`, `p1a`, `p1b_s`, `p1b`, `p1c`, `p1d` 全部設為 `⏭️`
- 意即：MarkItDown 輸出直接進 proofreader，跳過所有 PDF 專屬 phases
