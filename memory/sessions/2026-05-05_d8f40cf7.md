## 1. Implementation Plan

# 🚀 跨技能連動與參考資料架構設計 (Optimal Architecture)

針對您提出的三種情境，以及多重參考資料（課本、課件、文獻）的交叉比對需求，這涉及到系統層級的架構設計。以下是最優解的架構提案：

## 1. 應對三種資料情境的解法

我們必須保持 `audio_transcriber` 與 `doc_parser` 兩個技能的**「低耦合、高內聚」**，讓它們能獨立運作，也能互相支援：

| 情境 | 最佳處理流程 | 系統行為 |
|:---|:---|:---|
| **僅有錄音** | 只執行 `audio_transcriber` | Phase 2 找不到參考資料，直接依賴 LLM 的內建知識進行純文字校對。這不會造成系統錯誤，僅是沒有額外 Context。 |
| **僅有課件/文獻** | 只執行 `doc_parser` | 系統將 PDF 轉換為結構化的 `sanitized.md`、提取圖表，並作為知識庫儲存，不涉及語音流程。 |
| **有錄音 + 有課件** | 先執行 `doc_parser`，再執行 `audio_transcriber` | Phase 2 在校對時，會自動跨技能讀取 `doc_parser` 產生的 `sanitized.md` 講義作為 Context，大幅提升專有名詞準確率。 |

## 2. 混合型參考資料（課本、文獻、課件）如何交叉比對？

如果我們把一整本課本、三篇文獻、加上課件的文字「全部塞給」Phase 2 的大模型，**絕對會引發 OOM (記憶體耗盡) 或是超過 Context Window (上下文長度限制)**，而且雜訊太多會讓模型產生嚴重幻覺。

### 🌟 最優解架構：分層萃取與動態載入

這需要將參考資料分為「輕量級」與「重量級」兩種來處理：

#### A. 輕量級：全域詞庫 (Global Glossary)
- **概念**：不論是課本、文獻還是課件，只要丟進 `doc_parser`，系統就會在背景跑一個 NLP/LLM 任務，把裡面所有的**專有名詞、術語、英文縮寫**萃取出來，統一寫入該科目的 `Global Glossary`。
- **應用**：在語音校對時，這個詞庫（通常只有幾百個字）會**無條件、全部**塞進 Prompt 給模型參考。這能解決 80% 專有名詞聽錯的問題，且完全不佔用 Token。

#### B. 重量級：RAG-Lite (檢索增強校對)
- **概念**：對於長篇課本與文獻，不能全塞。當 Phase 2 在校對某個 5 分鐘的語音片段時，系統應該即時去搜尋 `doc_parser` 的輸出，找出**最相關的 2~3 段文獻內容**。
- **實作方式**：
  1. **前綴匹配 (目前的作法)**：只讀取與音檔同名的講義（例如 `L02-1.m4a` 優先讀取 `L02-1` 的 `sanitized.md` 的前 2 萬字）。適合簡報課件。
  2. **混合降級方案 (Fallback)**：
     - 首選讀取 `doc_parser` 產生的 Markdown。
     - 如果找不到，才退回使用 `pypdf` 直接從 PDF 抽文字（作為保底機制，容許使用者忘記跑 `doc_parser`）。

## Proposed Implementation (本期實作目標)

為了腳踏實地，我們這次修改 `audio_transcriber/scripts/phases/p02_proofread.py` 採取以下邏輯：

1. **保留容錯性 (Fallback)**：先去 `doc_parser` 找高品質的 Markdown；如果沒找到，保留原本 `pypdf` 直接抽 PDF 的機制（這樣即使使用者沒跑 `doc_parser`，系統也不會報錯）。
2. **多檔案整合**：如果該科目有多個文件，我們先整合出一個涵蓋範圍廣的 `Glossary`。但在全文參考上，我們僅抓取與音檔**檔名最相近**的文件（通常是上課簡報）作為主體。

## 2. Walkthrough / Summary

# 🛠️ 跨技能連動架構實作完成

我們已經順利將 `doc_parser` (文獻解析) 與 `audio_transcriber` (語音轉錄) 兩大技能的資源池打通！

## 1. 智慧尋找最佳參考資料 (Smart Resolution)
現在當 `audio_transcriber` 進入 Phase 2 (校對模式) 時，它的資料尋找邏輯升級為：
1. **跨技能尋找**：系統會自動穿越到 `data/doc_parser/output/01_processed/{科目}/` 底下尋找。
2. **檔名比對**：如果音檔叫 `L02-1.m4a`，它會先尋找 `L02-1` 的講義；若無，會自動退一步尋找共用的 `L02` 講義。
3. **讀取高品質 Markdown**：優先讀取經過去除頁首頁尾的完美版 `sanitized.md`；若無，讀取原始的 `raw_extracted.md`。

## 2. 容錯降級機制 (Graceful Fallback)
為了確保系統穩定性，我們保留了舊版的機制：
- 如果該科目完全沒有跑過 `doc_parser`，或者沒有產生對應的 Markdown。
- 系統會無縫**「降級 (Fallback)」**，回到 `audio_transcriber` 自己的目錄下尋找 `.pdf`，並使用 `pypdf` 抽取純文字作為備案。
- 這樣確保了無論您用哪種方式丟資料，系統都不會崩潰報錯。

## 3. 全新 CLI 提示
當您下次執行 `run_all.py` 時，注意看終端機的 Log，您會清楚知道大模型到底吃了什麼等級的參考資料：
- **最佳狀態**：`📚 [跨技能參考] 成功讀取 doc-parser 解析 Markdown: L02 (15000 字元)`
- **降級狀態**：`📚 [降級參考] 成功讀取原始 PDF 講義: ... (15000 字元)`

現在您的 Open-Claw 系統已經真正具備了**「先解析文獻，再完美校對語音」**的聯動工作流！
