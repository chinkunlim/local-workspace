## 區塊 A：記憶體修復策略

### 1. 防線 1：Ollama API 強制卸載 (Explicit Model Unloading)
`core/llm_client.py` 目前已具備 `unload_model` 函數，但並未在所有任務結束時被強制調用。
**實作策略**：我們將在各階段腳本（如 `p02_proofread.py`, `p03_merge.py`）的 `run()` 方法中實作清理機制，或者在 Pipeline Base 統一實作。當整個文本或批次處理完成後，強制呼叫卸載。
**Pseudo-code**：
```python
# 在 PipelineBase 或各階段的 run() 結束前呼叫
def run(self, ...):
    try:
        # ... 執行各項生成任務 ...
        response = self.llm_client.generate(model=model_name, prompt=prompt)
    finally:
        # 任務結束，強制釋放 VRAM / RAM
        self.log(f"🧹 階段結束，強制卸載模型 {model_name}...")
        self.llm_client.unload_model(model=model_name, logger=self.log)

# llm_client.py 內的實作邏輯 (現有邏輯，確保 timeout 足夠):
def unload_model(self, model: str, logger=None):
    requests.post(self.api_url, json={
        "model": model,
        "keep_alive": 0
    }, timeout=5)
```

### 2. 防線 2：嚴格的 Context 上限與併發控制
目前 `core/task_queue.py` 的 `LocalTaskQueue` 已實作了**絕對序列化 (Max Workers = 1)** 的佇列，確保不同時觸發多個子行程。
但各 Skill 的 `config.yaml` 設定的 `num_ctx` 過大（高達 32768），這是造成記憶體暴增的主因。
**將被修改的 `config.yaml` 參數配置**：
- **`skills/audio-transcriber/config/config.yaml`**：
  - Phase 2 (Proofread): `num_ctx` 從 `16384` 降至 `8192`
  - Phase 3 (Merge): `num_ctx` 從 `16384` 降至 `8192`
- **`skills/note_generator/config/config.yaml`**：
  - Extraction: `num_ctx` 從 `16384` 降至 `8192`
  - Synthesis: `num_ctx` 從 `32768` 降至 `8192`
- **`skills/doc-parser/config/config.yaml`**：
  - Parse/Extract: `num_ctx` 從 `16384` 降至 `8192`
*(備註：所有高於 8192 的 `num_ctx` 皆會被強制下修)*

### 3. 防線 3：復活 RAM Guard (System-level Protection)
將 `RAM_Safety_Guard.py` 的精華邏輯融入 `core/task_queue.py`，成為 Task Queue 在派發任務前的「閘門 (Gate)」。
**實作策略**：在 `_worker_loop` 取出任務後、執行 `subprocess.run` 之前，使用 `psutil` 偵測可用記憶體比例。
**Pseudo-code**：
```python
import psutil
import time

def _check_ram_and_wait(self):
    while True:
        vm = psutil.virtual_memory()
        free_ratio = vm.available / vm.total
        if free_ratio < 0.15: # 低於 15% 可用記憶體
            print(f"⚠️ [RAM Guard] 系統可用記憶體過低 ({free_ratio*100:.1f}%)。暫停 Task Queue 30 秒等待資源釋放...")
            time.sleep(30)
        else:
            break

# 整合進 task_queue.py:
def _worker_loop(self):
    while True:
        task = self.q.get()
        # [NEW] 在執行繁重生成任務前，強制檢查系統 RAM
        self._check_ram_and_wait()
        
        # 執行任務
        result = subprocess.run(cmd, cwd=cwd)
        # ...
```
