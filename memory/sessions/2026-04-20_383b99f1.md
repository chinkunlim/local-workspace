## 1. Implementation Plan

# Open Claw 深度體檢報告與重構藍圖

這是一份針對 `local-workspace` 進行靜態掃描後的分析報告與後續重構計畫。目標是建立防 OOM 的基礎設施、純化 Skill 職責，並實現與 Open WebUI 的完美雙向接力。

## 🔍 Phase 1: 專案深度盤點與分析 (Audit Report)

### 1. 總體架構與冗餘分析 (Architecture)
- **「幽靈」依賴**：雖然 `core/web_ui` 的實體檔案與路由已被刪除，但 `core/inbox_daemon.py` 仍會嘗試向 `http://127.0.0.1:18789` 發送 HTTP POST。當該 API 不存在時，才會退回到 `subprocess.Popen`。
- **過時的上下文**：`core/cli_runner.py` 等模組的註解仍停留在「為 Flask WebUI 服務」的設計思維中。系統需要全面轉向為純粹的本機 CLI 代理程式與 Task Queue 架構。

### 2. 程式邏輯與併發防護 (Logic & Compliance)
- **⚠️ 嚴重的 OOM 崩潰風險**：`inbox_daemon.py` 在 WebUI 不存在時，會**無限制地並發執行 `subprocess.Popen`**。一旦使用者丟入 10 個 PDF 檔，系統會同時啟動 10 個 `run_all.py`，導致 Ollama 載入多個模型實例，瞬間撐爆 16GB 記憶體並引發系統崩潰。
- **I/O 穩定性**：目前的 `atomic_writer.py` 採用 `tempfile.mkstemp` 與 `os.replace`，具備作業系統層級的 Crash-safe 防護。但在引入 Task Queue 後，單檔的原子寫入已經足夠安全，可支撐高頻寫入。

### 3. Skill 獨立性與純化分析 (Skill-by-Skill Audit)
- **防幻覺防線破洞**：
  - `audio-transcriber`：`phase0` 溫度為 `0.1`，`phase3` 甚至達 `0.2`。
  - `doc-parser`：`phase1d` (Vision) 溫度設定為 `0.1`。
  - **結論**：所有「萃取層 (Extraction)」的模型溫度必須強制作為 `0`，杜絕任何創造性幻覺。
- **職責越界**：
  - `audio-transcriber` 內部殘留了 `Phase 3: 合併與分段指令` (位於 `prompt.md`)。這類「整合加工」的任務應當轉交給 `note_generator`。`audio-transcriber` 應僅限於輸出精準校對過的純文本。
- **`note_generator` 溫度過高**：在執行 `Phase 5 Part A` (Map 提取知識點) 時，配置使用了 `0.4` 與 `0.6` 的溫度，這在「精確提取」的 Map 階段極易引發幻覺，溫度應針對 Map 與 Reduce 階段分別調整。

### 4. 內部資料流接力 (Internal Data Flow)
- 目前檔案的流轉路徑：`inbox/` -> SystemInboxDaemon -> `data/<skill>/input/<subject>`。
- **問題點**：萃取 (Extraction) 與 加工 (Processing) 的邊界模糊。`audio-transcriber` 處理完後沒有自動將產出導流給 `note_generator`。我們需要將流程標準化：Raw -> Extracted -> Processed -> Knowledge Base。

### 5. Open WebUI 雙向接力相容性 (External Handoff Audit)
- **[輸出接力 (Open Claw -> WebUI)]**：目前的輸出端點止於 `atomic_writer.py` 寫入 Markdown。缺少能將這些筆記自動切塊 (Chunking) 並推送到 ChromaDB 或 WebUI 知識庫的橋樑。
- **[輸入接力 (WebUI -> Open Claw)]**：`core/cli.py` 提供的 `--subject`, `--file`, `--force` 參數非常乾淨，完美契合 Open WebUI Custom Tool 的 `subprocess.run()` 呼叫。

## 🛠️ Phase 2: 架構重構藍圖 (Execution Plan)

> [!IMPORTANT]
> 此階段將徹底解決 OOM 問題，並建立與 Open WebUI 的雙向生態。

### 1. 基礎設施強化：實作防 OOM 的 Task Queue
- **廢除 HTTP 殘留**：移除 `inbox_daemon.py` 中對 WebUI API 的所有 `urllib` 請求。
- **導入 `core/task_queue.py`**：
  - 在背景建立一個單一的 Worker Thread。
  - 當 `inbox_daemon.py` 收到新檔案時，不再直接呼叫 `subprocess.Popen`，而是將 `(skill, filepath)` 加入 Python 的 `queue.Queue`。
  - Worker 依序取出任務執行，保證同時間只有一個 Ollama 任務在跑，徹底消滅 OOM 危機。

### 2. 純化與防幻覺：重構 Skills
- **參數鎖定**：修改所有萃取層 (`audio-transcriber`, `doc-parser`) 的 `config.yaml`，強制將 `temperature` 設定為 `0`。
- **Prompt 淨化**：移除 `audio-transcriber/config/prompt.md` 裡的「合併與分段」邏輯，確保它只做錯字校對。把排版、結構化任務集中到 `note_generator` 之中。

### 3. Open WebUI 雙向整合生態系 (The Next Level)
- **正向接力 (Knowledge Pusher)**：
  - 建立 `core/knowledge_pusher.py`。
  - 在 Pipeline 尾端攔截輸出的 Markdown 檔案。
  - 使用 HTTP POST 直接呼叫 Open WebUI 的 `/api/v1/knowledge` API（或利用現有的 `infra/open-webui` 工具），將整理好的知識入庫。
- **反向接力 (Custom Tooling)**：
  - 規劃一份 `infra/open-webui/custom_tools/open_claw_tool.py` 的實作範例。讓 WebUI 能透過 `subprocess` 喚起 `python3 core/cli_runner.py run_doc_parser --file <PDF>`。
- **Obsidian 反向觸發監控 (Watchdog)**：
  - 增強 `inbox_daemon.py`，讓它監聽 `data/*/output/` 的 Markdown 檔案。如果檔案的 YAML Frontmatter 被修改為 `status: rewrite`，就自動將檔案路徑塞入 Task Queue，由 `note_generator` 重新處理。

## 2. Walkthrough / Summary

# 🚀 Open Claw 終極盤點與重構完成 (Phase 3)

所有重構任務均已執行完畢。以下是本次升級的核心成果：

## 1. 🛡️ 基礎設施強化：防 OOM 任務佇列

> [!IMPORTANT]  
> 徹底消滅了因多檔並發導致的 Ollama OOM (Out of Memory) 崩潰危機。

- **新增 `core/task_queue.py`**：實作了輕量級、本機單線程的任務佇列 (`queue.Queue`)。
- **重構 `inbox_daemon.py`**：移除了殘留的 Flask WebUI HTTP POST 邏輯。現在當檔案進入 `data/raw/` 時，Daemon 會將任務送入 `task_queue`，由背景 Worker 依序、循序執行，確保同一時間只會有一個 Ollama 任務運行。
- **Obsidian 雙向觸發機制 (Watchdog)**：`inbox_daemon.py` 現在會監聽 `data/*/output/`。如果使用者在 Obsidian 內將筆記標頭改為 `status: rewrite`，Watchdog 會自動捕捉並將其排入 `note_generator` 佇列進行重新整理。

## 2. 🧊 純化與防幻覺防線

> [!TIP]  
> 模型幻覺通常發生在 Extraction 階段，我們已將其徹底封死。

- **溫度強制歸零**：
  - `audio-transcriber`, `doc-parser`, `smart_highlighter` 的 `config.yaml` 中，所有 LLM 參數的 `temperature` 已強制設定為 `0`。
  - `note_generator` 的 Map 階段溫度也下調至 `0.1`，以確保知識提取時不無中生有。
- **職責純化**：
  - 改寫了 `audio-transcriber` 的 `Phase 3` prompt，拔除所有排版與分段指令，確保它**只做無損合併**。所有高階的排版與筆記合成職責，正式移交給 `note_generator`。

## 3. 🔄 Open WebUI 雙向整合生態系

> [!NOTE]  
> Open Claw 現在不再是一個孤島，它可以與 Open WebUI 進行無縫雙向接力。

- **[正向接力] `core/knowledge_pusher.py`**：
  - 建立了一支推送腳本，可以在 Pipeline 結束後，將結構化筆記透過 API 推入 Open WebUI 的知識庫 (ChromaDB) 中，供未來的 RAG 使用。
- **[反向接力] `open_claw_tool.py`**：
  - 在 `infra/open-webui/custom_tools/` 提供了一個 Pydantic Tool 腳本範例。讓 Open WebUI 上的模型可以直接透過 `subprocess` 喚醒本機的 Open Claw CLI 進行 PDF 或語音處理。
